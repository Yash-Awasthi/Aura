# R&D-X — Swift Export Assessment for AuraCore KMP

**Status:** Research complete — implementation deferred to Kotlin 2.2.20-stable
**Assessed:** 2026-06-09
**Kotlin baseline:** 2.0.21 (current in `gradle/libs.versions.toml`)
**Target:** Kotlin 2.2.20 stable on AURA's build toolchain

---

## 1. Objective

Evaluate the feasibility of exposing `AuraCore` (the KMP shared module) to native
iOS Swift clients via Kotlin's new `@SwiftExport` mechanism, replacing the current
Objective-C interop layer.

The goal is a Swift-native AURA SDK surface: no `AuraKt.shared`, no `KotlinByteArray`,
no manual `NSData ↔ ByteArray` bridging.

---

## 2. Current iOS Integration (Baseline)

### Architecture today
```
Swift (iOS)
    └── AuraCore.xcframework  (compiled KMP)
            └── ObjC header (generated by Kotlin/Native)
                    └── AuraCoreKt.shared (Kotlin singleton bridge)
```

### Pain points with current ObjC interop

| Issue | Impact |
|-------|--------|
| All Kotlin types exposed as `AnyObject` | Unsafe casting in Swift, no autocomplete |
| `ByteArray` → `NSData` bridge is manual | 20+ lines of bridging per crypto call |
| Kotlin `sealed class` → ObjC protocol | Loses exhaustive `switch` in Swift |
| `Flow<T>` not exposed | Real-time updates require callback polling |
| `suspend fun` → callback hell | Async crypto ops awkward in Swift |
| `Pair<A,B>` → `__AuraCorePair` | Unreadable in Xcode |
| No Swift `throws` mapping | Kotlin exceptions become `NSError` opaque blobs |

Estimated bridging code: **~1,200 lines** of `AuraBridge.swift` manually maintained.

---

## 3. Kotlin Swift Export (`@SwiftExport`) Overview

Swift Export (KMP) generates a `.swiftmodule` directly from Kotlin sources,
bypassing the ObjC header layer entirely. Available since Kotlin 2.0.20 (experimental),
targeting stable in Kotlin 2.2.x.

### Key capabilities in 2.0.20–2.1.x

| Kotlin construct | Swift Export output | ObjC (current) |
|-----------------|---------------------|-----------------|
| `data class` | Swift `struct` (value type) | ObjC `@interface` (ref type) |
| `sealed class` | Swift `enum` with associated values | ObjC `protocol` |
| `suspend fun` | `async throws func` | Callback-based |
| `Flow<T>` | `AsyncSequence<T>` | Manual KotlinFlow wrapper |
| `ByteArray` | `[UInt8]` / `Data` | `NSData` |
| `Result<T,E>` | `Result<T,E>` | Opaque `NSError` |
| `companion object` | Swift `static` members | `AuraCoreKt.companion` |

### Key limitations in 2.0.20–2.1.x (must resolve before adoption)

1. **Generics**: Generic Kotlin classes are not yet fully supported. `HybridKemEngine<T>`
   cannot be exported. Workaround: specialise for `ByteArray` variants.

2. **Kotlin/Native memory model**: Swift Export still requires Kotlin/Native (not Kotlin/JVM).
   AuraCore's crypto uses BouncyCastle — BouncyCastle for Kotlin/Native requires
   `bcprov-jdk18on` recompiled for `iosArm64`. Current `bcprov-jdk18on:1.79` is JVM-only.
   **Blocker** until BouncyCastle ships an official KMP/Native artifact or we use
   an alternative (e.g., `kotlinx-crypto` or `swift-crypto` via C interop).

3. **`@ObjCName` vs `@SwiftExport` conflicts**: Classes annotated with `@ObjCName`
   may produce symbol name conflicts when Swift Export is enabled simultaneously.
   All `@ObjCName` annotations in AuraCore must be audited.

4. **Framework size**: Swift Export produces a separate `.swiftmodule`; combined with
   the `.xcframework`, the total iOS binary grows ~15–20%. Acceptable for AURA's use case.

5. **Kotlin 2.2.20 requirement**: The stable release is projected for Q3 2026.
   Using experimental Swift Export on 2.0/2.1 risks breaking changes between
   Kotlin minor versions.

## 4. AuraCore Symbol Audit

### Symbols that export cleanly (no changes needed)

```kotlin
// These map 1:1 to Swift
data class AuraContactCard(val did: String, val displayName: String, val publicKey: ByteArray)
sealed class ExchangeResult { data class Success(...); data class Error(...) }
object AuraCrypto { fun sign(data: ByteArray): ByteArray }
suspend fun NearbyExchangeService.initiateExchange(card: AuraContactCard): ExchangeResult
```

### Symbols requiring adaptation for Swift Export

| Symbol | Issue | Recommended fix |
|--------|-------|-----------------|
| `HybridIdentityKey` | Uses `DilithiumPrivateKeyParameters` (BouncyCastle JVM type) | Wrap behind `AuraIdentityKeyFacade` with `ByteArray`-only API |
| `DIDCommTransport` | `Flow<ByteArray>` — needs `AsyncSequence` mapping | No code change needed; Swift Export handles Flow in 2.2+ |
| `PrivacyPassClient` | `RSABlindedEngine` (BouncyCastle) | Same wrapper strategy as HybridIdentityKey |
| `AuditSigningCoordinator` | `ECPrivateKey` (JVM type) leaks into public API | Replace with `ByteArray` parameter in public function |
| `MdocDocument` | `X509Certificate` (JVM) in return type | Return `ByteArray` (DER) + parse in Swift layer |
| `VcIssuer` | `X509Certificate` issuer param | Same: pass DER bytes |
| `BusinessCardImporter` | Android `Context` constructor param | Expect / actual with iOS `NSBundle` equivalent |

### @ObjCName annotations to audit

Search: `grep -r "@ObjCName" app/src/main --include="*.kt"`
Current count: **0** (none declared — no conflicts expected on initial Swift Export enable).

## 5. Migration Plan

### Phase 0 — Prerequisites (before Kotlin 2.2.20)

- [ ] Resolve BouncyCastle KMP/Native blocker (Q3 2026)
  - Option A: Wait for official BouncyCastle KMP artifact
  - Option B: Replace BouncyCastle usage in `commonMain` with `kotlinx-crypto` or C interop
  - Option C: Keep JVM BouncyCastle in `androidMain`; use Swift Crypto (`CryptoKit`) in `iosMain`
- [ ] Add `iosMain` source set to `AuraCore` module (currently Android-only)
- [ ] Wrap all JVM-specific types behind `expect`/`actual` interfaces

### Phase 1 — Enable Swift Export (Kotlin 2.2.20-stable)

```kotlin
// settings.gradle.kts
kotlin {
    iosArm64 {
        binaries.framework("AuraCore") {
            isStatic = true
        }
    }
    swiftExport {
        // Opt-in per https://kotlinlang.org/docs/swift-export.html
        enableSwiftExport = true
    }
}
```

### Phase 2 — API surface cleanup

- Replace `ByteArray` ↔ `[UInt8]` in 7 identified symbols (see table above)
- Annotate `@ShouldRefineInSwift` on Kotlin functions that benefit from Swift refinement
- Add `@SwiftName` annotations for idiomatic Swift naming:
  ```kotlin
  @SwiftName("initiateExchange(with:)")
  suspend fun NearbyExchangeService.initiateExchange(card: AuraContactCard): ExchangeResult
  ```

### Phase 3 — Remove ObjC bridge layer

- Delete `AuraBridge.swift` (~1,200 lines)
- Update `AuraApp/iOS/Package.swift` to consume `.swiftmodule` directly
- Run Xcode build; fix any remaining symbol resolution issues

### Phase 4 — Validate

- Unit tests: `XCTest` suite compiles and passes against Swift-exported AuraCore
- Integration: AURA iOS prototype completes a BLE exchange end-to-end
- Binary size: verify < 20% increase in `.ipa` size

## 7. Decision

**Defer Swift Export adoption to Kotlin 2.2.20-stable (projected Q3 2026).**

Using experimental Swift Export on 2.0/2.1 introduces unacceptable migration risk
(breaking API changes between Kotlin minor versions). The BouncyCastle JVM blocker
must also be resolved before any `iosMain` source set can use cryptographic primitives.

**Immediate action items** (pre-Kotlin 2.2.20):
1. Create `AuraCryptoFacade` interface in `commonMain` that exposes only `ByteArray`-based
   API, wrapping all BouncyCastle / JVM types in `androidMain` actual implementations.
2. Add `iosMain` source set stub (no-op `actual` implementations returning empty results)
   so the module structure is ready when Swift Export stabilises.
3. Track Kotlin 2.2.x changelog for Swift Export GA announcement.
